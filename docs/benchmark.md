# MCPツール ベンチマーク比較レポート

**計測日:** 2026-05-22  
**環境:** WSL2 (Linux 6.6, AMD64), Rust 1.x  
**対象:** 自作 MCP Tools (Rust) vs 標準 Bash コマンド

---

## 概要

このレポートは、自作 MCPサーバー（`mcp-tools`）の各ツールについて、  
Claude Code の Bash ツールで同等コマンドを実行した場合と比較し、  
**実行速度** と **コンテキスト消費トークン数** の差を定量評価したものです。

ベンチマークは3つのシナリオで実施しました：

| シナリオ | 内容 |
|----------|------|
| **A: 小規模** | `mcp-tools/src/`（5ファイル）を対象とした基本操作 |
| **B: 大規模** | `~/.cargo/registry`（612ファイル, 14,000+マッチ）を対象とした検索 |
| **C: ウォームサーバー** | MCP永続プロセス vs Bash 都度プロセス起動（10回連続実行） |

---

| 出力バイト数 | 289 | 289 | 1.00x |
| 概算トークン数 | 72 | 72 | 1.00x |

### A-3: diff（main.rs vs grep.rs）

| 指標 | MCP | Bash | Bash/MCP |
|------|----:|-----:|--------:|
| 平均実行時間 (ms) | 1.8 | 1.7 | 0.97x |

| 出力バイト数 | 13,142 | 13,232 | 1.01x |

| 指標 | MCP | Bash | Bash/MCP |

**小規模まとめ:** 速度・トークン量ともにほぼ同等（±30%以内）。  
MCPサーバー側でのプロセス起動コストが Bash コマンドと同程度のため差が出にくい。

## シナリオB: 大規模操作（★ 最大の差が出るシナリオ）

対象: `~/.cargo/registry/src/`（Rust クレート 612ファイル, 14,000+ マッチ）

### B-1: grep（`fn ` パターン）

> MCP grep は `max_matches=100`（デフォルト）で自動打ち切り  
> Bash grep は全 14,594 マッチを返却

| 指標 | MCP | Bash | Bash/MCP |
|------|----:|-----:|--------:|
| 平均実行時間 (ms) | **4.0** | 40.3 | **10.10x** |
| 最小実行時間 (ms) | **3.5** | 37.9 | **10.74x** |
| 出力バイト数 | **16,139** | 2,371,358 | **146.93x** |
| 概算トークン数 | **4,034** | 592,839 | **146.96x** |

→ **速度 10倍、トークン消費 147倍削減**

### B-2: find（.rs ファイル列挙）

> MCP find は `max_results=200`（デフォルト）で自動打ち切り  
> Bash find は全 612 件を返却

| 指標 | MCP | Bash | Bash/MCP |
|------|----:|-----:|--------:|
| 平均実行時間 (ms) | **2.0** | 3.2 | **1.62x** |
| 最小実行時間 (ms) | **1.4** | 2.9 | **2.08x** |
| 出力バイト数 | **21,542** | 66,291 | **3.08x** |
| 概算トークン数 | **5,385** | 16,572 | **3.08x** |

→ **速度 1.6倍、トークン消費 3倍削減**

## シナリオC: ウォームサーバー（連続10リクエスト）

> Claude Code でMCPサーバーを使う場合、サーバープロセスは  
> セッション開始時に1回起動し、全リクエストを処理し続ける（ウォームサーバー）。  
> Bash ツールは毎回シェルプロセスを新規起動する。

| 指標 | MCP（ウォーム） | Bash（都度起動） |
|------|---------------:|---------------:|
| 1リクエストあたり平均 (ms) | **0.26** | 1.40 |
| 10リクエスト合計 (ms) | **2.6** | 14.0 |
| 速度改善 | — | **5.34x 遅い** |

→ **実際の Claude Code セッションでは、MCP ツールは1リクエストあたり約5倍高速**

## 考察

### なぜ小規模では差が出ないのか

MCPサーバーのベンチマークでは1リクエストごとにバイナリを再起動しているため、  
Rust バイナリの起動コスト（約1ms）が純粋な処理時間を上回ってしまう。  
実際の Claude Code セッションではサーバーは常時起動しているため、  
シナリオCが実運用に近い比較になる。

### なぜ大規模では劇的な差が出るのか

MCPツールには `max_matches`（grep: デフォルト100件）と  
`max_results`（find: デフォルト200件）による**自動打ち切り**が組み込まれている。

Bash の `grep` は全マッチを返すため、大規模コードベースでは：
- **14,594 件のマッチ → 2.37MB の出力 → 約 593,000 トークン**

対してMCP grepは：
- **100件で打ち切り → 16KB の出力 → 約 4,034 トークン**

コンテキストウィンドウの節約が最大の効果であり、  
これはコスト削減とモデルの応答品質向上に直結する。

### diff・git_diff について

`diff` と `git_diff` は出力量がそのまま差分全体に依存するため、  
打ち切りによるトークン節約効果は得られない。  
ただし `context` パラメータで周辺行数を絞ることはできる。

## 結論

| 用途 | 推奨 | 理由 |
|------|------|------|
| 大規模コードベースの検索（grep/find） | **MCPツール** | トークン147倍削減、速度10倍以上 |
| 小規模ファイルの操作 | どちらでも可 | 実質的な差なし |
| 連続した複数の検索操作 | **MCPツール** | ウォームサーバーで5倍高速 |
| git diff・ファイル差分 | どちらでも可 | 出力量に差なし |

**最大のメリットは大規模コードベースでの検索時のトークン削減。**  
セッション中に何度も検索を繰り返す場合は、ウォームサーバー効果も加わり  
コスト・速度の両面で優位性が出る。
